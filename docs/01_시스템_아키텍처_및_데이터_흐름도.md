# 시스템 아키텍처 및 데이터 흐름도 (Luxury-Jung)

## 1. 시스템 개요
'(이)력서리-정 (Luxury-Jung)'은 다양한 형태의 이력서(Word, HWP, Excel, PDF 등)를 업로드받아 텍스트를 추출하고, AI를 활용해 정형화된 JSON 데이터로 변환 후 데이터베이스에 저장 및 조회할 수 있도록 하는 통합 웹 시스템입니다.

## 2. 시스템 아키텍처

```mermaid
graph TD
    %% Frontend
    Client[Web Browser\n(React/Next.js)]
    
    %% API Gateway / Main Backend
    subgraph Main Web Application (Java Spring Boot)
        AuthService[인증/인가 서비스]
        SearchService[조회/페이징 서비스]
        UploadService[파일 업로드 대기 서비스]
    end
    
    %% File Parser & AI Worker
    subgraph Parsing Agent (Python FastAPI / Celery)
        FileExtractor[포맷별 텍스트 추출기]
        AIProcessor[AI API 연동 및 정제 로직]
    end
    
    %% Storage
    subgraph Data Layer
        RDBMS[(PostgreSQL\n(메인 데이터))]
        SearchEngine[(Elasticsearch\n(대용량 검색/필터링))]
        ObjectStorage[(S3 / Local Storage\n(원본 파일))]
    end
    
    %% Connections
    Client -- "조회/업로드 API" --> MainWebApplication
    MainWebApplication -- "원본 파일 저장" --> ObjectStorage
    UploadService -- "추출 작업 큐전송" --> ParsingAgent
    FileExtractor -- "파일 다운로드" --> ObjectStorage
    FileExtractor -- "텍스트 파싱" --> AIProcessor
    AIProcessor -- "LLM API 호출" --> ExternalAI[OpenAI / Gemini API]
    AIProcessor -- "정형화된 JSON 저장" --> RDBMS
    RDBMS -- "데이터 동기화" --> SearchEngine
    SearchService -- "빠른 조회/검색" --> SearchEngine
```

## 3. 핵심 아키텍처 요소
- **Main Web Application (사용자 응대 및 조회 전담):**
  - 회원 관리(로그인, 권한 부여), 이력서 조회 로직, 파일 업로드의 진입점을 담당합니다.
- **Parsing Agent (비동기 파일 처리 전담):**
  - 이력서 파싱은 시간이 많이 소요되고 CPU 의존적인 작업입니다. 따라서 Python 기반의 별도 워커로 분리하여 웹 서비스의 지연(Blocking)을 방지합니다.
- **데이터 저장소 분리 전략 (CQRS 패턴 고려):**
  - 원본/관계형 데이터 저장은 RDBMS(PostgreSQL)가 담당하고, 대용량 데이터의 빠르고 복잡한 조건 검색(형태소 분석 등)은 Elasticsearch가 담당하여 지연시간을 최소화합니다.

## 4. 데이터 흐름도 (Data Flow)
1. **관리자 파일 업로드 Flow:**
   - 관리자(Admin) 로그인 -> 이력서 파일(복수) 업로드 -> Main App이 스토리지에 파일 저장 -> 메시지 큐(RabbitMQ/Redis)에 작업 발행 -> Parsing Agent가 작업 수신 -> 텍스트 추출 -> AI 정제 -> RDBMS에 Insert -> Search Engine에 동기화 완료.
2. **이력서 조회 Flow:**
   - 인가된 사용자 로그인 -> 검색 조건 입력(경력, 기술 스택 등) -> Search App에서 Elasticsearch 인덱스 조회 -> 커서 기반 또는 오프셋 기반 페이징 형태로 클라이언트 응답 반환.
