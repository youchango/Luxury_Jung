# 테스트 시나리오 및 품질 검증서

## 1. 목적
Luxury-Jung 프로젝트의 핵심 기능인 '다양한 포맷의 파일 파싱/정제' 및 '대용량 데이터의 저지연 검색'의 안정성을 보장합니다.

## 2. 테스트 환경
- 단위 테스트: `JUnit 5` (메인 웹 앱), `pytest` (파싱 에이전트)
- 통합/부하 테스트: `JMeter` 또는 `k6`
- 모의 AI: API Mocking 처리.

## 3. 기능 테스트 시나리오

### [TC-1] 관리자 엑셀/HWP/PDF 다중 파일 업로드 및 분석
- **사전조건:** 관리자 계정 로그인 상태
- **절차:** 10개의 서로 다른 포맷(Word, PDF, HWP 혼합)의 이력서를 드래그 앤 드롭 업로드.
- **예상 결과:** 모든 파일이 Storage에 등록되며, Parsing 상태가 'Processing'으로 표시됨. 잠시 후 성공적으로 처리된 파일은 'Completed', 텍스트가 전혀 없는 스캔 기반 PDF는 'Failed' 등으로 명확히 상태가 업데이트 됨.

### [TC-2] AI 분석 결과 데이터 무결성 검증
- **절차:** 파싱 완료 후 DB에 적재된 데이터를 확인.
- **예상 결과:** 지정된 속성(`name`, `birthdate`, `techStacks` 등)의 필드 타입이 정확하며(예: 연차가 Number로 적재), 누락 정보는 null로 일관되게 처리됨.

### [TC-3] 조건부 이력서 검색 및 권한 검증
- **절차:** 비로그인 사용자 및 일반 사용자 / 관리자 각각 검색 API 접근.
- **예상 결과:** 인가된 회원(Login User)만 결과 목록을 수신하고, 비로그인은 403 Forbidden 반환. 
- **검색 절차:** 연차 '3년~5년', 기술 스택 'Java, Spring' 필터 설정 후 검색.
- **예상 결과:** 해당 조건에 정확히 부합하는 지원자의 레코드 반환.

## 4. 비기능 분질 검증 (성능)

### [Perf-1] 대용량 DB 조회 지연시간 최소화
- **사전조건:** 데이터베이스에 모의(Mock) 이력서 데이터 10만 건 이상 적재.
- **절차:** 필터 조건(5가지 이상 중복 조건)을 포함하여 1페이지 조회. 다시 맨 끝 페이지(예: 1000페이지) 조회.
- **결과 목표:** API 응답 속도 P95(최악의 경우 95%) 200ms 이내 보장.
(쿼리 플랜 검토 방안: offset 기반 페이징의 성능 저하시 인덱스 보강 혹은 No-Offset(Cursor) 기반 페이징 적용, 또는 Elasticsearch 적용 평가)

### [Perf-2] 동시 처리 스루풋
- **절차:** 관리자 5명이 동시에 50건의 파일을 병렬로 업로드.
- **결과 목표:** API Gateway가 차단되지 않아야 하며 큐 매커니즘에 의해 워커가 무리하지 않고 순차적/병렬적으로 처리 완료. (메모리 OOM 에러 방지)
